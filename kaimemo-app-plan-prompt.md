# 【最重要制約】

1. AWS無料枠内で収まる設計を最優先すること
2. 段階的実装（MVP → 機能追加）とすること
3. 各フェーズごとに「実機確認 → 問題なければ次へ進む」方式
4. 環境構築は必要なタイミングで段階的に行う
5. 初期段階で過剰なインフラを作らない
6. 将来的にスケール可能な構造にする
7. アナリティクスは今回は未実装

# 【AWS無料枠制限条件】

- EC2: t2.micro / t3.micro 750時間以内
- RDS: db.t3.micro 750時間以内
- S3: 5GB以内
- データ転送：無料枠内
- Cognito: 無料枠内ユーザー数想定（少人数）
- CloudWatch最小利用
- NAT Gatewayは使わない（コスト回避）
- 可能なら単一AZ構成（初期）
- WebホスティングはS3 + CloudFront（AWS無料枠・無料利用枠内）を採用すること
- AWS Amplify Hostingは使わない

無料枠を超える可能性がある箇所は必ず明示すること。

## グループ共有

- 招待リンク方式

## リスト共有方法

リストを外部に共有する際は以下の3方法から選択できる。
共有ボタンをタップするとシートが開き、方法を選択する（シンプルなボトムシート1枚）。

- **閲覧専用リンク**: サインアップ不要で誰でも閲覧可能なURL発行。家族への共有に最適
  - リンクはサーバーで生成・DB保存。有効期限は設定可（デフォルト無期限）
  - 閲覧者はチェック操作不可（読み取り専用）
- **LINE共有**: リスト内容を整形したテキストとしてLINEアプリに渡す
  - Webは `line://msg/text/` スキームでLINEアプリ起動、Mobileは `expo-sharing` 経由
  - サインアップ不要、LINEアカウントのみで送信可能
- **テキストコピー**: リスト内容をクリップボードにコピー
  - どのアプリにも貼り付け可能な汎用形式

## アイテム管理

- 担当者割り当て

- 店舗動線順ソート（カテゴリ設定ベースのソート、AI不要）

- カテゴリ並び順カスタマイズ（設定画面でドラッグ変更）
- 検索・フィルタ機能（リスト内アイテムを絞り込み）

## 通知

- 時間指定リマインダー

- 位置情報ジオフェンス通知（スマホのみ・Expo TaskManager使用）

# 【UI設計原則】

## 1タップ操作 vs 隠し機能の分離

機能を多く持つが、**画面上に常時表示するのは日常操作に限定する**。
カスタマイズや設定は必要な時だけ辿り着ける場所に配置する。

### 常時表示（1タップでアクセス）

| 画面 | 1タップ操作 |
|------|------------|
| ホーム | リスト選択・新規リスト作成 |
| リスト | アイテム追加（テキスト）・チェック完了・音声入力ボタン |
| リスト | 共有ボタン（→ボトムシートで3方法選択） |
| リスト | 検索バー |

### 2タップ（メニュー・スワイプ経由）

| 操作 | アクセス方法 |
|------|------------|
| アイテム編集（数量・メモ・写真・カテゴリ変更） | アイテムをタップ → 詳細シート |
| マイリストから一括追加 | [+ 追加] → [マイリストから追加] |
| 定期リマインダー設定 | アイテム詳細 → リマインダー設定 |
| リスト削除・名前変更 | リスト長押し or 右上メニュー |

### 設定画面に隠す（日常的には触らない）

- カテゴリ並び順カスタマイズ（ドラッグ変更）
- カテゴリ色マッピング（将来的な変更用）
- マイリスト管理（作成・編集・削除）
- 閲覧専用リンクの有効期限設定
- 通知設定（時間指定リマインダー、ジオフェンス設定）
- AI買い忘れ提案のON/OFF

### 実装時の注意

- **デフォルト状態はシンプルに保つ**: 初期画面には設定項目を一切露出しない
- **プログレッシブディスクロージャー**: 詳細はタップして初めて出現する
- **カスタマイズ機能は「設定」タブに集約**: ナビゲーションバーに設定アイコンを1つ置くだけ
- 設定を変更していないユーザーはデフォルト値が最適動作するよう設計する

## UIを作り込むべきタイミング（出戻り防止ガイド）

後のフェーズで「やっぱりここから作り直し」になりやすいポイントを先に整理する。
UIの作り込みは「そのフェーズの機能を実装する直前」が最適。

### ① Phase 5開始前に必ず確立する（デザイン基盤）

Phase 5以降で画面・コンポーネントが急増するため、**ここで基盤を固めないと全画面の修正が必要になる**。

- **ナビゲーション構造の確定**
  - ボトムナビのタブ構成を決める（例: リスト / グループ / 設定 の3タブ）
  - ルーティング設計はこの時点で完成させる。後から変えると全画面に影響する

- **カラーパレット・タイポグラフィ・スペーシングの定数化**
  - `colors.ts`, `spacing.ts` などのデザイントークンをコードに定義
  - 後で「色を変えたい」となっても1ファイル変更で済む構造にする

- **共通コンポーネントの作成**（Phase 6以降で使い回す）
  - `BottomSheet`: 音声入力・共有・アイテム詳細で共通利用
  - `Modal`: 確認ダイアログ等
  - `ListItem`: アイテムカード（**Phase 6で拡張するため、拡張しやすい構造で作る**）
  - `SearchBar`: 検索バー

- **アイテムカードの拡張設計**（最重要）
  - 現状: 商品名 + チェックボックスのみ
  - Phase 6以降に追加: カテゴリ色ライン + 数量/単位 + メモインジケーター + 写真サムネイル
  - **Phase 5開始時点で「将来の追加を想定した余白」を持つカード構造にしておく**
  - そうしないとPhase 6でカード全体を作り直すことになる

```
アイテムカード構造（Phase 5時点で設計、Phase 6で埋める）:

|色| [  商品名  ] [数量・単位] [メモ📝] [写真🖼]  ☑
     ↑           ↑                              ↑
  Phase 6追加   Phase 6追加                  既存
```

### ② Phase 6で必ず作り込む（差別化UIの中核）

Phase 6はこのアプリの**差別化機能が集中するフェーズ**。ここのUIの質が評価を決める。
実装効率より**UX品質を優先**してよい唯一のフェーズ。

- **音声入力モーダル** — アプリの看板機能。雑に作るとユーザーが使わなくなる
  - 録音中のアニメーション（波形 or パルス）を必ず付ける
  - 候補リストのチェック操作は1タップで即反応すること（遅延NG）

- **アイテムカード完成版** — カテゴリ色ライン + 数量 + メモ表示
  - カード左端の色ラインはCSSのみで実装（画像不要）
  - 情報密度が上がるが、視覚的にシンプルに見えるよう余白・フォントサイズを調整

- **検索バー** — 全リスト画面の最上部に固定配置
  - 入力中はリアルタイムでフィルタされること（ボタン押下不要）

### ③ 後回しにしてよいUI（Phase 7以降）

- 設定画面のデザイン（機能が確定してから）
- マイリスト管理画面（設定画面の一部として後から追加）
- AI買い忘れ提案のバッジ・バナーデザイン
- ダークモード対応（Phase 8以降の余裕があるときに）

### ④ 絶対に触らないUI（機能完成まで手をつけない）

- アニメーション・画面遷移エフェクト（機能が全部動いてから最後に調整）
- ブランディング（ロゴ・アプリアイコン・スプラッシュ画面）
- マーケティング用スクリーンショット向けのビジュアル調整

# 【AI音声入力・買い物支援機能 詳細仕様】

## 概要

以下の機能によって「入力の簡略化・買い忘れ防止・買い物効率向上」を実現する。
AIを活用するが、**UIは可能な限りシンプル**に保つこと。

## 音声入力（AI強化版）

- キャプチャ: Web Speech API（Web）/ Expo Speech Recognition（Mobile）
- 音声ストリーミング認識 → 途中テキスト生成 → AI解析 → 商品候補リアルタイム表示
- **AI補正**:
  - 言い間違い補正（例：ぎゅうにゅう → 牛乳）
  - 聞き取りミス補正（例：トマロ → トマト）
  - 文脈補完（例：「カレーの材料」発話 → 玉ねぎ/にんじん/じゃがいも/カレールー を候補生成）
  - **ユーザー履歴を考慮した文脈補完**: 同じフレーズで過去に追加・削除した実績をAIへの入力に含め、提案精度を向上させる（後述）

## マイリスト（定期購入テンプレート）

ユーザーが事前に「よく買うセット」を登録しておき、買い物リストに一括追加できる機能。

- ユーザーはマイリストを複数作成可能（例：「週の定番」「月1まとめ買い」「鍋セット」）
- 各マイリストにアイテムを登録（名前・数量・単位も設定可）
- 買い物リストの「追加」メニューから「マイリストから追加」を選ぶと一覧が表示される
- 選択したマイリストの全アイテムを買い物リストに一括追加
- 既にリストに存在するアイテムは重複追加しない（スキップ or 数量加算）

### DBスキーマ

```sql
-- my_lists テーブル（新規）
id          String    -- PK
user_id     String    -- FK to users
name        String    -- リスト名（例: "週の定番"）
created_at  DateTime

-- my_list_items テーブル（新規）
id          String   -- PK
my_list_id  String   -- FK to my_lists
name        String   -- 商品名
quantity    Decimal? -- 数量
unit        String?  -- 単位
sort_order  Int      -- 表示順
```

### UIフロー

```
買い物リスト画面 → [+ 追加] → [マイリストから追加]
  ↓
マイリスト一覧（カード選択）
  ↓
「週の定番（7件）を追加する」ボタン
  ↓
一括追加完了
```

## 数量・単位指定

- 各アイテムに数量（数値）と単位（個・本・g・kg・ml・L・袋・パック など）を設定可能
- 入力はオプション。未入力でも登録可能
- DBの `items` テーブルに `quantity`（Decimal?）・`unit`（String?）カラムを追加
- リスト表示: 商品名の右に「2本」「500g」のように小さく表示
- 音声入力でも「牛乳 2本」と発話すれば数量・単位を自動解析してセット

## 候補選択UI（シンプル）

```
🎤 録音中  話してください

☑ 牛乳
☑ 卵
☑ トマト

[終了]  [追加する]
```

- モーダル1枚のチェックリスト形式（既存UIスタイルを踏襲）
- 候補は初期状態で全選択（☑）
- タップで除外（☑ → ☐）
- 「追加する」ボタンで選択分のみリストへ一括登録

## 商品カテゴリ分類

- AIが商品名からカテゴリを自動判定（初回のみ）
- カテゴリ一覧（デフォルト）: 野菜・肉・魚・乳製品・パン・冷凍・日用品・その他
- DBの `items` テーブルに `category` カラムを追加
- **ユーザーカテゴリ上書き優先ルール**:
  - ユーザーがAI提案カテゴリを手動で変更した場合、その「商品名→カテゴリ」の対応をDBに保存
  - 次回同じ商品名が登録された場合はAIを呼ばず、保存済みのユーザー設定を使用
  - 保存先: `user_item_preferences` テーブル（`user_id`, `item_name`, `category`）

## 店舗動線順ソート

- カテゴリが決まればAI不要。ユーザー設定の並び順に従って単純ソートするのみ
- デフォルト動線順: 野菜→肉→魚→乳製品→パン→冷凍→日用品
- ユーザーはカテゴリ並び順を設定画面でドラッグ変更可能
- 並び順はDBのユーザー設定（`user_settings` テーブル）に保存
- ソートロジックはサーバーまたはクライアントの配列ソートのみ（AIコール不要）

## カテゴリ色表示

- リスト表示時にカテゴリごとに左端カラーラインを表示（CSSで実装）
- 色は固定（アイコンは使わない）

| カテゴリ | 色 |
|---------|-----|
| 野菜    | #4CAF50（緑） |
| 肉      | #F44336（赤） |
| 魚      | #2196F3（青） |
| 乳製品  | #00BCD4（水色） |
| パン    | #FF9800（オレンジ） |
| 冷凍    | #9C27B0（紫） |
| 日用品  | #607D8B（グレー） |
| その他  | #9E9E9E（グレー） |

## 検索・フィルタ機能

リスト内のアイテムが増えても素早く目的の商品を見つけられるようにする。
UIはシンプルに保ち、検索バー1本で完結させる。

- **テキスト検索**: 商品名の部分一致（フロントエンドのみで完結、APIコール不要）
- **カテゴリフィルタ**: カテゴリ選択タブまたはドロップダウン（1タップ切替）
- **状態フィルタ**: 未チェックのみ表示 / チェック済みのみ表示 / すべて表示
- UIイメージ:
  ```
  [🔍 検索...         ] [野菜▼] [未チェックのみ]
  ```
- フィルタ状態はページ内のローカルステートで管理（DB保存不要）

## 定期リマインダー

- 商品単位で購入周期設定（例：牛乳→3日ごと / トイレットペーパー→月1回）
- DBの `items` テーブルに `remind_interval_days` カラムを追加
- 設定周期に応じてプッシュ通知（Expo Push Notifications）

## AI買い忘れ提案

- 購入履歴（チェック完了ログ）から購入頻度・周期・最終購入日を分析
- 購入タイミングが近い商品を画面上でバッジ表示 or プッシュ通知で提案
- 例：「そろそろ牛乳を買いませんか？（前回購入から3日経過）」

### 分析対象

- 購入頻度（何日ごとに購入しているか）
- 最終購入日
- カテゴリ
- 定期リマインダー設定の有無

## AI API連携設計（Phase 6実装時）

### リクエスト仕様

```
POST /api/ai/parse-voice
  body: {
    text: "カレーの材料",
    locale: "ja",
    user_context: {
      // 同フレーズの過去履歴（最小限）
      phrase_history: [
        {
          phrase: "カレーの材料",
          added:   ["玉ねぎ", "にんじん", "じゃがいも"],  // 承認して追加したもの
          removed: ["カレールー"]                          // チェック外して除外したもの
        }
      ]
    }
  }
  response: {
    items: [
      { name: "玉ねぎ",    category: "野菜",   selected: true },
      { name: "にんじん",  category: "野菜",   selected: true },
      { name: "じゃがいも",category: "野菜",   selected: true },
      { name: "カレールー",category: "その他", selected: false } // 過去除外→初期チェックOFF
    ]
  }
```

### 処理フロー

```
[ユーザーの発話テキスト確定]
    ↓
サーバー: user_item_preferences から同商品名のカテゴリ上書き設定を取得
サーバー: item_phrase_history から同フレーズの追加・除外履歴を取得（直近5回分）
    ↓
Claude API呼び出し（上記履歴を最小限のコンテキストとして渡す）
    ↓
レスポンス生成（過去除外品は selected: false で返す）
    ↓
カテゴリは user_item_preferences に登録済みならAI判定を上書き
```

- サーバーサイドでClaude APIを呼び出し（APIキーをEC2環境変数に保存）
- クライアントは `/api/ai/parse-voice` のみ呼び出す（APIキーをクライアントに渡さない）
